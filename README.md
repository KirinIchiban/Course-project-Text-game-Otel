# Отель «У Погибшего Альпиниста» - Интерактивная текстовая игра на F#

## О проекте

Курсовой проект - интерактивная текстовая игра «Отель "У Погибшего Альпиниста"», реализованная на языке F# с использованием функциональной парадигмы.

Цель проекта - продемонстрировать моделирование игрового мира через неизменяемое состояние, монадическую композицию и асинхронные вычисления без императивных конструкций, глобальных переменных и изменяемых объектов.

## Структура проекта

| Файл | Назначение |

## Как запустить

### Требования

- .NET 8.0 SDK или выше
- Git

### Клонирование репозитория

```bash
git clone https://github.com/KirinIchiban/Course-project-Text-game-Otel
cd Course-project-Text-game-Otel/project
```

### Запуск игры

```bash
dotnet run --project build.fsproj
```

### Автоматическое прохождение

Скрипт `run_game.sh` генерирует файл `commands.txt` со списком команд, подаёт его на стандартный ввод программе через `cat commands.txt | dotnet run --project build.fsproj`, вывод перенаправляется в `output.txt`.

```bash
chmod +x run_game.sh
./run_game.sh
```

После выполнения результат будет находиться в файле `output.txt`.

## Сюжет

Протагонист - Петер Глебски, инспектор полиции, находящийся в отпуске. Он прибывает в удалённый горный отель «У Погибшего Альпиниста» с целью отдыха. Вскоре в отель является таинственный однорукий незнакомец, представившийся Луарвиком Л. Луарвиком, который разыскивает постояльца по имени Олаф. Через некоторое время Олафа обнаруживают мёртвым в его номере. Глебски вынужден начать расследование: опросить постояльцев, собрать улики, проанализировать факты и принять финальное решение, определяющее не только судьбу присутствующих, но и исход первого контакта с внеземным разумом.

## Архитектура

### Состояние игры (GameState)

Состояние игры представляет собой запись `GameState`, определённый в модуле `HotelGame.Types` файла `types.fs`. Запись содержит все данные, описывающие текущий момент игры. Состояние является неизменяемым (immutable): любое изменение создаёт новый экземпляр записи с обновлёнными полями; старый экземпляр остаётся нетронутым. Это исключает ошибки, связанные с непреднамеренным изменением состояния из разных частей программы.

`GameState` состоит из следующих полей:

| Поле | Тип | Назначение |
| :--- | :--- | :--- |

| `Doors` | `(string * string) list` | Список пар локаций, между которыми возможен переход. Каждая пара `(a, b)` означает, что из `a` можно перейти в `b` и наоборот. |

### Монада Dialog<'a>

Монада `Dialog<'a>` является основным способом построить "умную" последовательность операций в проекте без копирования повторяющихся строк кода. Она определена в модуле `HotelGame.Mechanics` файла `mechanics.fs`:

```fsharp
type Dialog<'a> = GameState -> Async<'a * GameState>
```

`Dialog<'a>` представляет собой функцию, которая принимает на вход текущее состояние игры `GameState`, возвращает асинхронное вычисление `Async`, результатом которого является кортеж `('a * GameState)`, где `'a` - полезная нагрузка операции, `GameState` - новое состояние игры после выполнения операции.

Параметр типа `'a` является обобщённым и принимает различные типы в зависимости от конкретной операции:

- `unit` - для операций, которые изменяют состояние, но не возвращают значимого результата. Пример: функция `look` (осмотр локации), `saveGame` (сохранение игры).
- `string` - для операций, возвращающих текстовое описание. Пример: функция `examineItem` в случае успешного осмотра, `roofDummyEvent`.
- `Result<unit, string>` - для операций, которые могут завершиться ошибкой. Успех обозначается `Ok unit`, ошибка - `Error string` с описанием. Пример: `goto`, `takeItem`, `openDoor`.
- `Result<string, string>` - для операций, возвращающих текст или ошибку. Пример: `talkTo`, `interrogate`, `examineItem`.

Монада `Dialog<'a>` позволяет описывать игровые действия как значения, которые можно комбинировать друг с другом, не выполняя их немедленно. Фактическое выполнение (включая ввод-вывод) происходит только при запуске всей цепочки вычислений из точки входа.

### DialogBuilder - построитель вычислительного выражения

`DialogBuilder` - это класс конструктор computation expression  `Dialog<'a>`. 

Методы `DialogBuilder` и соответствующие им синтаксические конструкции:

| Ключевое слово | Метод | Сигнатура | Назначение |
| :--- | :--- | :--- | :--- |
| `let! x = m` | `Bind` | `Dialog<'a> -> ('a -> Dialog<'b>) -> Dialog<'b>` | Выполняет монадическое действие `m`, извлекает из него значение `x` типа `'a` и передаёт его в функцию-продолжение. Одновременно передаёт новое состояние в продолжение. |
| `do! m` | `Bind` (через `Zero`) | Аналогично `Bind`, но результат `m` игнорируется (ожидается `Dialog<unit>`) | Выполняет действие `m`, не извлекая значения. Используется для операций с побочными эффектами. |
| `return x` | `Return` | `'a -> Dialog<'a>` | Оборачивает значение `x` в монаду, возвращая действие, которое возвращает `x` и не меняет состояние. |
| `return! m` | `ReturnFrom` | `Dialog<'a> -> Dialog<'a>` | Выполняет монадическое действие `m` и продолжает цепочку как его результат. Используется для встраивания одной монады в другую. |
| `try..with` | `TryWith` | `Dialog<'a> -> (exn -> Dialog<'a>) -> Dialog<'a>` | Обрабатывает исключения: выполняет `body`, в случае исключения вызывает `handler`. |
| `for x in xs do` | `For` | `seq<'a> -> ('a -> Dialog<unit>) -> Dialog<unit>` | Последовательно выполняет действие `body` для каждого элемента последовательности, передавая состояние по цепочке. |
| (неявно) | `Delay` | `(unit -> Dialog<'a>) -> Dialog<'a>` | Откладывает выполнение переданного вычисления. Принимает лямбду и возвращает её без немедленного выполнения. |
| (неявно) | `Combine` | `Dialog<unit> -> Dialog<'a> -> Dialog<'a>` | Связывает два действия: выполняет первое (игнорируя результат), затем выполняет второе. |

#### Bind - основной метод монады

Метод `Bind` является основным связующим механизмом монады. Его определение:

```fsharp
let bind (m: Dialog<'a>) (f: 'a -> Dialog<'b>) : Dialog<'b> =
    fun state -> async {
        let! (a, newState) = m state
        return! f a newState
    }
```
Bind берёт значение `a` из первой операции `m` и передаёт его на вход функции-продолжению `f`. При этом производится дополнительная обработка: функция `f` получает новое состояние `newState`, а не исходное состояние `state`. Это обеспечивает сквозную передачу состояния через всю цепочку вычислений.


Пример использования `Bind` через computation expression:

```fsharp
dialog {
    let! state = getState       
    let! result = goto "Холл"     
    do! writeLine "Перемещение выполнено" 
    return result                  
}
```

Компилятор преобразует этот код в цепочку вызовов `Bind`, `Return` и других методов `DialogBuilder`.

### Специальные операторы

В проекте определены три пользовательских оператора для компактной записи монадических композиций. Все они определены в файле `mechanics.fs`.

#### Оператор `>>=` (монадическое связывание)

```fsharp
let (>>=) = bind
```

Оператор `>>=` является синонимом функции `bind`. Запись `m >>= f` эквивалентна записи `bind m f`. Используется для построения цепочек операций без использования computation expression.

Пример из `mechanics.fs` с скомпозицией:

```fsharp
let updateState (updater: GameState -> GameState) : Dialog<unit> =
    getState >>= (updater >> setState)
```

#### Оператор `>>!` (связывание с обработкой Result)

```fsharp
let (>>!) action valid = handleResult action valid
```

Оператор `>>!` предназначен для работы с монадическими действиями, возвращающими `Result<'a, 'e>`. Он выполняет действие `action`, и:

- если результат - `Ok value`, вызывает функцию `valid` с аргументом `value`;
- если результат - `Error err`, выводит сообщение об ошибке через `writeLine err` и пропускает вызов функции `valid`. В этом случае цепочка продолжается без выполнения последующих действий, зависящих от успеха.

Пример из `gameloop.fs` для операций, возвращающих текст:

```fsharp
do! examineItem item >>! (fun text -> writeLine text)
```

Если `examineItem item` вернул `Ok text`, текст выводится на экран. Если вернул `Error err`, выводится ошибка.

#### Оператор `>>.` (последовательное выполнение)

```fsharp
let (>>.) action next = handleUnit action next
```

Сигнатура: `Dialog<unit> -> Dialog<'a> -> Dialog<'a>`.

Оператор `>>.` предназначен для последовательного выполнения двух монадических действий. Он выполняет первое действие `action`, игнорирует его результат (ожидается `unit`) и затем выполняет второе действие `next`.

Пример из `plot_functions.fs`:

```fsharp
do! triggerEvents place >>. autoSave
```

Эквивалентно:

```fsharp
do! triggerEvents place
do! autoSave
```

Оператор `>>.` явно показывает, что результат первого действия не используется во втором. 
### Игровой цикл и Delay

#### gameLoop

Главный игровой цикл представляет собой рекурсивную функцию, определённую в файле `gameloop.fs`:

Функция `gameLoop` работает следующим образом:

1. Выводит приглашение ко вводу `"> "`.
2. Считывает строку ввода от пользователя.
3. Разбивает строку на части по пробелам.
4. Сопоставляет части с известными командами.
5. Вызывает соответствующую операцию.
6. Вызывает `return! gameLoop ()` для перехода к следующей итерации цикла.

Строка `return! gameLoop ()`. `return!` в computation expression описывает действие: «выполни эту монаду и продолжи цепочку как её результат». Поскольку `gameLoop` возвращает `Dialog<unit>`, `return!` выполняет новый экземпляр `gameLoop` и встраивает его результат в текущую цепочку. Можно скахать, рекурсивно описывается цикл, в котором каждый новый шаг добавляется в цепочку асинхронных вычислений.

#### Delay

Метод `Delay` в классе `DialogBuilder` определён следующим образом:

```fsharp
member _.Delay(f) = f()
```

Сигнатура: `(unit -> Dialog<'a>) -> Dialog<'a>`.

Функция `Delay` гарантирует, что тело `dialog { ... }` не выполняется в момент объявления выражения. Вместо этого выполнение откладывается до момента, когда до выражения дойдёт очередь в цепочке `Bind`.

Без `Delay` рекурсивный вызов `gameLoop()` в строке `return! gameLoop ()` произошёл бы немедленно в момент определения функции, что привело бы к бесконечной рекурсии и переполнению стека.

С `Delay` процесс работает так:

1. Компилятор превращает тело `dialog { ... }` в функцию, которая пока не выполняется.
2. `Delay` принимает эту функцию и не запускает её сразу, а откладывает на потом.
3. `gameLoop ()` возвращает не результат, а действие, готовое принять состояние.
4. Когда действию передают состояние, тогда начинается работа ввод-вывод.
5. Дойдя до `return! gameLoop ()`, создаётся новое отложенное действие, и круг повторяется без накопления стека.


## Особенности и подсказки

Сюжетные события активируются автоматически при первом входе в определённые локации. Активация происходит в функции `triggerEvents` файла `movements.fs`, которая вызывает соответствующие функции из `plot_functions.fs`.
- Улики записываются автоматически при осмотре соответствующих предметов или локаций. Добавление улик происходит в функциях `examineBodyEvent`, `roofDummyEvent`, `hinkleFoundEvent`, `examineBagEvent`, `examineDummy`, а также при осмотре записки.


**Финал**: команда `закончить отпустить` приводит к концовке «Отпустить», где игрок помогает инопланетянам бежать. Команда `закончить задержать` приводит к концовке «Задержать», где игрок следует закону, что приводит к гибели инопланетян. Для активации финала требуется, чтобы все улики были собраны.
